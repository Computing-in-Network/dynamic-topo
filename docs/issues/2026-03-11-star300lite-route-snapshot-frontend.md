# star300lite 静态路由快照前后端联动展示

## 背景

当前 `star300lite` 300 节点环境已经完成了静态路由集中式下发：

- `dynamic-topo` 通过 WebSocket 推送实时拓扑
- `push_static_routes.py` 根据当前拓扑计算最短跳第一跳，并批量写入 300 个节点容器
- 前端已经可以基于实时 `links` 在页面上本地重算一条路径并做可视化展示

但这一版前端展示存在一个偏差：

- 页面展示的是“浏览器根据当前拓扑重新算出的路径”
- 不是“控制器最近一次真正已经下发到容器里的路由结果”

在拓扑变化频繁、控制器存在稳定帧门槛和最小应用间隔的情况下，这两者可能不完全一致。

## 目标

让前端展示基于控制器“最近一次成功下发”的真实静态路由结果，而不是前端自行重算，从而保证展示内容和容器内实际路由状态一致。

## 交付范围

1. 路由控制器快照输出
   - `push_static_routes.py` 在每次成功下发后写出只读 JSON 快照
   - 快照包含：
     - 节点元信息
     - 每个源节点到每个目的节点的已下发第一跳
     - 连通分量大小
     - 最近一次下发帧号、边数、更新时间

2. WS 只读接口
   - `stream_server` 增加 `route_snapshot` 控制动作
   - 前端可通过现有 WebSocket 控制通道直接获取最近一次已下发路由快照

3. 前端展示切换
   - 静态路由演示面板改为读取已下发快照
   - 展示：
     - 快照状态
     - 快照时间
     - 已下发帧号
     - 连通分量大小
     - 跳数
     - 第一跳
     - 完整路径
   - 球面路径高亮改为基于快照恢复结果

4. 启动脚本联动
   - `start_star300lite_control_plane.sh` 统一给 `push_static_routes.py` 和 `stream_server` 传递快照路径

5. 首轮下发改进
   - 路由控制器首轮不再等待多帧稳定后才下发
   - 首轮先生成第一份快照，后续更新继续保留稳定帧门槛

## 验收标准

1. `run/star300lite/route_snapshot.json` 能随控制器成功下发而更新
2. 通过 WebSocket 发送 `route_snapshot` 动作可返回有效快照
3. 前端静态路由演示面板显示的数据来自快照，而不是浏览器本地重算
4. 抽样节点容器中的 `ip route` 结果与前端展示的第一跳一致

## 说明

这一改动的重点不是“再做一套算路逻辑”，而是把展示层和真实控制面状态对齐，确保汇报和演示看到的是“已经下发到节点里的结果”。
